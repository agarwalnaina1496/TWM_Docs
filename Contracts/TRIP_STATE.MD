# Trip State

## Purpose

Trip State is the canonical representation of a single trip within TWM.

It acts as the shared state layer between Trip Matcher, Trip Planner, and future TWM features.

---

## Design Principles

### Single Trip Scope

A Trip State object represents one trip only.

Multiple trips for the same traveler must have separate Trip State records.

### Progressive Discovery

Trip State reflects only what is currently known about the trip.

The system does not create placeholder fields for information not yet discovered.

The only exceptions are `trip_context.required_inputs` — which pre-defines the six fields Matcher needs — and top-level lifecycle fields (`status`, `stage`).

### Separation of Concerns

Trip State is split into four distinct objects. Each has a clear owner and lifecycle:

| Object | Owner | Lifecycle |
|---|---|---|
| `trip_context` | The trip | Created at start, lives until trip is done. Read by both Matcher and Planner. |
| `matcher_state` | Trip Matcher | Active during matching. Not touched by Planner. |
| `planner_state` | Trip Planner | Null until destination confirmed. Not touched by Matcher. |
| `traveler_profile` | The user | Not in TripState. Managed separately in Phase 3+. |

### Two-Zone Model

**Free zone** — `status: free` — everything is exploratory and reversible.

**Committed zone** — `status: committed` — triggered by payment. Stage moves forward only.

## Schema

```json
{
  "trip_id": "trip_7f3a9c",
  "status": "free",
  "stage": "matching",

  "trip_context": {
    "required_inputs": {
      "origin_city": null,
      "budget": null,
      "budget_unit": null,
      "duration_nights": null,
      "num_travelers": null,
      "travel_month": null
    },
    "preferences": {},
    "selected_option": null
  },

  "matcher_state": {
    "generate_ready": false,
    "conversation_context": {
      "last_scout_message": null,
      "awaiting": null
    },
    "last_recommendations": null
  },

  "planner_state": null
}
```

## Field Definitions

### trip_id

Unique identifier for this trip. Generated by the UI when a new Trip State is created.

For MVP with localStorage: short UUID. When DB is introduced: primary key.

---

### status

Which zone the trip is in.

```
free        →  Exploratory. Everything reversible.
committed   →  Payment made. Stage moves forward only. Never goes back.
```

---

### stage

Current position in the trip lifecycle.

**Free zone (reversible):**
```
new        →  Trip State created. Nothing done yet.
matching   →  Scout collecting inputs.
ready      →  Scout signalled ready. Generate button visible.
matched    →  Destination or circuit confirmed.
planning   →  Planner started. Plan not complete.
planned    →  Full trip plan complete.
```

**Committed zone (forward only — triggered by payment):**
```
booked     →  Trip booked.
done       →  Trip completed.
```

`destination_confirmed` is not a separate field. Derived from `stage === "matched"`.

---

### trip_context

Shared context for the entire trip. Read by both Matcher and Planner. Written primarily during Matcher but carries forward into Planner.

#### trip_context.required_inputs

The six fields Meridian requires. Always present, initially null. The only pre-defined nulls in the schema.

```json
{
  "origin_city": "Bengaluru",
  "budget": 30000,
  "budget_unit": "total",
  "duration_nights": 3,
  "num_travelers": 3,
  "travel_month": "September"
}
```

`budget_unit` is required alongside `budget`. `50,000 total` for 3 people is very different from `50,000 per person`. Allowed values: `total` | `per_person`.

#### trip_context.preferences

Dynamic object. Everything known about this trip beyond the six required inputs.

Only discovered information. No predefined schema. No null fields.

Scalar preferences include a `confidence` score:

```
explicit   →  Traveler stated it directly
high       →  Strongly implied from context
medium     →  Reasonably inferred
low        →  Tentative signal
```

```json
{
  "trip_goal": {
    "value": "Bachelorette",
    "confidence": "explicit"
  },
  "crowd_tolerance": {
    "value": "low",
    "confidence": "high"
  },
  "walkability": {
    "value": "high",
    "confidence": "medium"
  },
  "group_type": "friends",
  "travel_style": ["relaxed", "social"],
  "nuanced_preferences": [
    "Wants a chill celebration vibe, not a big party scene",
    "Prefers boutique or aesthetic stays over standard hotels"
  ]
}
```

`travel_style`, `nuanced_preferences` — arrays, no confidence wrapper.
`group_type` and simple categoricals — plain values.

**Why preferences lives in trip_context and not matcher_state:**
Planner uses these too. Crowd tolerance, travel style, nuanced preferences all inform what Planner recommends for hotels, activities, pacing. They belong to the trip, not the matching workflow.

#### trip_context.selected_option

Set when traveler confirms a destination or circuit. Initially `null`.

```json
{ "type": "destination", "id": "pondicherry_puducherry" }
```

```json
{ "type": "circuit", "id": "rajasthan_classic" }
```

Lives in `trip_context` not `matcher_state` because it is the output of Matcher that Planner consumes. It survives Matcher's lifecycle.

---

### matcher_state

Matcher-only state. Planner does not read or write this.

#### matcher_state.generate_ready

`true` when Scout has collected all required inputs and asked "anything else?" — signalling the conversation has reached a natural handoff point.

UI shows "Generate Recommendations" button when `true`. User always pulls the trigger. Resets to `false` if user sends another message after Scout signals ready.

#### matcher_state.conversation_context

Carries just enough context for Scout to resume a conversation without history.

```json
{
  "last_scout_message": "How many nights are you thinking?",
  "awaiting": "duration_nights"
}
```

`last_scout_message` — the last message Scout sent. Scout reads this on the next turn to know what it said and avoid repeating itself.

`awaiting` — what Scout was waiting for. Simple string label. `null` when nothing specific is awaited.

This is not a conversation summary. Everything meaningful extracted from conversation is already in `trip_context`. `conversation_context` only carries what is needed to maintain conversational continuity — where Scout left off.

#### matcher_state.last_recommendations

The full Meridian output from the most recent generation. Initially `null`.

```json
{
  "options": [
    "pondicherry_puducherry",
    "coorg_karnataka",
    "gokarna_karnataka"
  ],
  "best_match": "pondicherry_puducherry",
  "generated_at": "2025-09-15T10:45:00Z",
  "meridian_output": { }
}
```

Stored so that on page refresh, Scout can re-engage with "you were looking at these three" using actual data — without calling Meridian again.

`meridian_output` is the full Meridian JSON. Heavy object — acceptable for localStorage at MVP scale. Moves server-side naturally when DB is introduced.

---

### planner_state

`null` until destination is confirmed and user enters Trip Planner.

Populated when Planner starts:

```json
{
  "conversation_context": {
    "last_message": "Which part of Pondicherry would you prefer to stay in?",
    "awaiting": "accommodation_area"
  },
  "preferences": {
    "accommodation_area": "White Town",
    "transport_preference": "scooter rental",
    "meal_preference": "local restaurants over hotel food"
  },
  "itinerary": null
}
```

`planner_state.preferences` — planning-specific preferences that emerge during the planning conversation. Separate from `trip_context.preferences` — these are more granular and specific to execution, not trip-level signals.

`planner_state.itinerary` — the output of Trip Planner when complete.

---

### traveler_profile

Not in TripState.

Persistent user-level preferences (recurring patterns, learned behaviour, travel history) are managed separately in Phase 3+ when user accounts are introduced.

At that point, the backend combines `TripState + TravelerProfile` before calling Meridian or the Planner agent. TripState never needs to know about it.

## Stage Transition Reference

| Action | Stage becomes |
|---|---|
| Trip State created | `new` |
| Scout starts collecting inputs | `matching` |
| User sends message after `ready` | `matching` |
| Scout signals ready | `ready` |
| User taps Generate, sees recommendations | `ready` (stage unchanged — not yet confirmed) |
| Destination confirmed | `matched` |
| User enters destination directly | `matched` |
| User returns to Matcher from Planner | `matching` |
| Planner started | `planning` |
| Plan complete | `planned` |
| Payment made | `booked` + `status → committed` |
| Trip completed | `done` |

---

## Storage

**MVP:** localStorage. One active trip at a time. Device-bound, session-persistent. No conversation history stored — only TripState.

**Post-MVP:** Database. Trip State becomes user-bound, cross-device, persistent across sessions. Multiple trips per user. Schema does not change — only the storage layer does.
